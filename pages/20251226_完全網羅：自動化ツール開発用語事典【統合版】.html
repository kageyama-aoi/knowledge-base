このドキュメントは、Pythonによる業務自動化ツール開発で使われる 設計思想・GUI構築・設定管理・Selenium操作を、 「初心者が他人に説明できるレベル」まで引き上げることを目的とした 再利用可能ナレッジ集です。

単なるコード解説ではなく、
なぜその設計が必要なのか／なぜその書き方をするのかを 言葉として理解できる構成にしています。

全体像・考え方

自動化ツール開発は「とりあえず動くコード」を書く段階から、 壊れにくく・変更に強く・説明できる構造へ進化させる必要があります。

本資料では以下を一貫した軸として扱います。

  • 処理を「役割」で分離する
  • 直接ライブラリを触らない
  • 設定と処理を混ぜない
  • GUIは状態で制御する

これらを支えているのが、オブジェクト指向・疎結合・抽象化といった考え方です。

前提条件
  • Pythonの基本文法（関数・クラス）を一度は見たことがある
  • 辞書（dict）とfor文の意味が分かる
  • SeleniumやTkinterは「聞いたことがある」レベルでOK

理解できない単語は、後半の用語事典で必ず回収できる構成です。

設計思想①：抽象クラスと疎結合

結論から言うと、「実行する側は中身を知らなくていい」状態を作るためです。

抽象基底クラス（ABC）は、 「このクラスを使うなら、最低限これだけは実装せよ」 という約束をコードで表現します。 


from abc import ABC, abstractmethod

class BaseHandler(ABC):
    @abstractmethod
    def execute(self):
        pass

これにより、呼び出し側は 「executeがあること」だけを信頼すればよくなります。

結果として、処理の差し替え・追加・削除が安全になります。

設計思想②：ユーティリティ化とラッパー関数

ライブラリを直接呼ぶコードが散らばると、 変更点が全体に波及します。

そこで「意味のある名前」を付けた関数で包みます。 


def input_text(driver, attribute, element, value):
    find_element(driver, attribute, element).send_keys(value)

これにより、

  • 処理内容が読める
  • 内部実装を隠せる
  • 修正箇所が1か所に集約される

これがカプセル化です。

GUI設計：動的生成と状態制御

GUIを固定レイアウトで作ると、 設定項目が増えた瞬間に破綻します。

設定データをループで回し、 UIをデータから生成します。 


for key, value in data.items():
    Label(...)
    Entry(...)

さらに、選択状態に応じて

  • 有効（normal）
  • 無効（disabled）

を切り替えることで、 「操作できない理由」をUIで表現できます。

設定管理：YAMLとラウンドトリップ

設定ファイルは人間が読むものです。

ruamel.yamlを使うことで、

  • コメント
  • インデント
  • クォート

を壊さずに保存できます。

これは「編集できる設定ファイル」を守るための重要な技術です。

Selenium：JSクリックと待機

画面操作が失敗する最大の理由は 「要素がまだ存在しない」か「物理クリック不可」です。

JavaScriptクリックは、 DOMに直接命令します。 


driver.execute_script("arguments[0].click();", element)

これは最後の切り札として覚えておく価値があります。

よくある詰まりポイント
  • クラスが増えて全体像が見えなくなる
  • 設定と処理が混ざって読めなくなる
  • GUIのstate制御を忘れて操作不能になる
  • Seleniumが「なぜか失敗する」

これらはすべて 設計不足ではなく、概念不足が原因です。

理解度チェック
  • 抽象クラスを使う理由を説明できるか
  • 疎結合とは「何が疎」なのか言えるか
  • ラッパー関数のメリットを他人に説明できるか
  • JSクリックを使う場面を想像できるか
