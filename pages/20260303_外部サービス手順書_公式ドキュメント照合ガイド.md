# 外部サービス手順書 — 公式ドキュメント照合ガイド

## このファイルの目的

外部サービス（Supabase, Vercel, GitHub Actions など）のセットアップ手順をAIに書いてもらう際、
UIの変更やキー名の変更によって手順がズレるケースがある。
このガイドは「AIに公式ドキュメントを確認させてから書かせる」ためのワークフローをまとめたもの。

---

## なぜ必要か

AIの学習データには締め切り（知識カットオフ）があり、
外部サービスのUI・用語・変数名は頻繁に変更される。

### 実際に起きた例（Supabase, 2025〜2026）

| 旧表記（AIが知っていた） | 現行表記（実際のUI） |
|---|---|
| `anon key` | `Publishable key`（形式: `sb_publishable_...`） |
| `service_role` | `Secret key`（形式: `sb_secret_...`） |
| Settings > API | Project Settings > API Keys |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` |

AIが古い情報をそのまま書いてしまい、手順通りに進めても「該当箇所が見つからない」状態になった。

---

## 解決策：AIに公式ドキュメントを確認させる

### 指示の書き方テンプレート

```
○○（サービス名）のセットアップ手順を書いて。
公式ドキュメントを確認してから書いてください。
特に：
- URLの取得場所
- APIキーの取得場所と名称
- 環境変数名
```

たったこれだけで、AIが `WebSearch` + `WebFetch` で公式ドキュメントを取得してから手順を作成する。

---

## 照合フロー

```
1. ユーザーが「公式ドキュメントを確認して書いて」と指示
        ↓
2. AIが WebSearch で最新ドキュメントを検索
        ↓
3. AIが WebFetch で公式サイトの内容を取得・読み込み
        ↓
4. 取得した情報をもとに手順書を作成
        ↓
5. 照合結果（差異があれば表で提示）を添えてユーザーに報告
        ↓
6. ユーザーが実際のUIと照合して最終確認
```

---

## 照合結果の報告フォーマット

手順書を作成した後、以下の形式で差異を報告する：

```markdown
## 公式ドキュメント照合結果

| 項目 | 旧記述 | 公式ドキュメント | 判定 |
|------|--------|------------------|------|
| キー名 | anon key | Publishable key | ❌ 修正済み |
| 取得場所 | Settings > API | Project Settings > API Keys | ❌ 修正済み |
| 環境変数名 | SUPABASE_ANON_KEY | SUPABASE_PUBLISHABLE_KEY | ❌ 修正済み |

Sources:
- [公式ドキュメントURL](https://...)
```

---

## チェックリスト：手順書レビュー時の確認項目

外部サービスの手順書を使う前に以下を確認する：

- [ ] ダッシュボードのメニュー名・場所が一致しているか
- [ ] APIキーの名称が現行UIと一致しているか
- [ ] 環境変数名がフレームワークの公式クイックスタートと一致しているか
- [ ] 「Sources」として公式ドキュメントのURLが記載されているか

---

## よく変更が起きるポイント

| カテゴリ | 変更が起きやすい内容 |
|---|---|
| ダッシュボードUI | メニュー名・階層・ボタンの場所 |
| APIキー名称 | anon → Publishable など命名規則の刷新 |
| 環境変数名 | フレームワーク推奨の変数名変更 |
| 認証フロー | OAuth設定の場所・パラメータ名 |
| デプロイ設定 | Vercel/Cloudflare の設定項目・UIの変更 |
