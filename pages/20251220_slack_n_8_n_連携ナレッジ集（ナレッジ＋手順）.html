ナレッジ編：Slack × n8n 連携を理解するための背景知識 #

Why（判断軸） 構造理解 再現性
ナレッジ① Slack 連携は「Slackアプリ開発」である

Slack と外部ツールの連携は「API設定」ではなく「アプリ開発」扱いです。Slack は外部からの操作をすべて Slack App 単位で管理し、人も Bot も同じユーザー概念の延長として扱います。したがって「Webhook を貼るだけ」「トークンを1個取って終わり」という設計ではありません。

結論として、n8n で Slack を使う＝Slack App を設計する行為です。

この前提を知らないと起きること

  • 設定の多さに混乱する
  • Scope / Token / Channel の関係が整理できない
  • 画面を行ったり来たりして迷子になる

ナレッジの使いどころ

  • 「どの画面が本丸か」を見失わない
  • 設計の単位を App として捉え直せる
ナレッジ② Slack API 管理画面（Web）は唯一の正解ソース

Slack API 管理画面（Web）は、Slack連携における一次情報です。権限・状態・制約が集約されているのは Web 管理画面だけで、n8n 側は Slack の状態を十分説明してくれません。

行動原則として、連携でおかしくなったら必ず Web 管理画面へ戻るを持つと迷子を防げます。

ナレッジ③ OAuth Scope は「できることリスト」ではない

OAuth Scope は「API を呼ぶ権利」であり、動作保証ではありません。たとえば channels:read があっても投稿できるとは限らず、実際の可否はチャンネル参加状態や Workspace 設定などにも依存します。

Scope は入口の鍵であって、通行証ではありません。

この認識がないと

  • 「Scope あるのに動かない」で詰まる
  • 原因を n8n 側に求めて遠回りする
  • 本当の原因（Slack 側ルール）に辿り着けない
ナレッジ④ Token は「権限のスナップショット」

Token は Scope 設定を含んだスナップショットです。Scope を変えても Token は自動更新されず、再インストール時に初めて再生成されます。

したがって、Scope を触ったら Token も作り直すが基本ルールになります。

ナレッジ⑤ Bot は「自動投稿用のユーザー」

Bot は特別な存在ではなく、制限付きユーザーとして扱われます。よって、チャンネル参加が必要で、参加していない場所には投稿できません。

not_in_channel は「あなたはこの場にいませんよ」という Slack の思想どおりのメッセージです。

ナレッジ⑥ n8n は「実行前チェックが強い」

Message Text が空などの必須入力不足は、Slack に投げる前に n8n が止めます。これは連携失敗ではなく、入力不足エラーです。

切り分けとして、n8n エラーか Slack エラーかをまず分けると早く解決できます。

ナレッジ⑦ Schedule Trigger は「条件 OR 実行」

Schedule Trigger は複数ルールを並べると、AND ではなく ORで動きます。テスト用・本番用のルールを同居させると想定外の時間に動くリスクがあります。

ナレッジ⑧ Activate は「本番スイッチ」

Execute と Activate は別物です。Execute はその場のテスト、Activate は常駐実行（スケジュール登録）です。手動で動いたのに自動で動かない場合、Activate 忘れが典型です。

補足App を複数持つか問題
  • 基本：用途ごとに App を分ける（通知用／監視用／社内ツール用 など）
  • 理由：権限の盛りすぎを防ぎ、影響範囲を小さくし、用途の説明がしやすい
  • 既存 App を使うのは「目的が明確」「権限が近い」場合に限る

実作業サマリー（ナレッジ対応・手順抽出版） #

以下は上のナレッジを踏まえたうえで、実際にやる操作だけを抜き出したもの。各手順の番号はナレッジ番号（①〜⑧）と対応。

【①②】 Slack API 管理画面へアクセス
  • Web ブラウザで https://api.slack.com/apps を開く
  • 「Your Apps」を開く
  • 対象 Workspace を確認
【①】 Slack App を作成 or 選択

新規

  • 「Create New App」→「From scratch」
  • App 名と Workspace を指定

既存

  • 一覧から対象 App を選択
【③】 OAuth & Permissions を設定
  • 左メニュー「OAuth & Permissions」を開く
  • Bot Token Scopes に追加：chat:write, channels:read
  • 必要に応じて：groups:read（プライベート）, im:read（DM）
【④】 Workspace へインストール（再インストール）
  • 「Install to Workspace」または「Reinstall to Workspace」
  • Slack 側で許可
【④⑤】 Bot User OAuth Token を取得
  • 表示される Bot User OAuth Token をコピー（xoxb- から始まる）
【⑤】 n8n に Slack Credential を登録
  • n8n の Credentials で Slack を新規作成
  • Connect using：Access Token
  • Bot Token（xoxb-…）を貼り付けて保存
【⑥】 Slack ノードを設定
  • Resource：Message
  • Operation：Send
  • Send Message To：Channel
  • Channel：送信先を選択／指定
  • Message Text：固定文 or Expression（例 {{$json.message}}）
【⑤⑥】 Bot をチャンネルに参加させる
  • Slack の対象チャンネルで /invite @Bot名
  • Bot がメンバーになったことを確認
【⑥】 動作確認（手動実行）
  • n8n で Execute step
  • Slack に投稿されることを確認
【⑦】 Schedule Trigger を設定（任意）
  • Trigger Interval を設定（例：Minutes / 5）
  • 不要ルールは削除（OR 実行事故防止）
【⑧】 ワークフローを Activate
  • ワークフロー全体を Activate（本番稼働）
  • 以後、スケジュール通り自動実行

典型エラーと意味（迷子防止） #

チャンネル一覧が出ない

  • 例：Could not load list
  • 例：missing required OAuth Scopes

原因：Scope 不足（channels:read など）／再インストール未実施。

対応：【③】【④】を見直す。

Slack error: not_in_channel

原因：Bot がそのチャンネルに参加していない。

対応：【⑤⑥】/invite @Bot名 を実行。

Problem executing workflow

  • 例：The workflow has issues and cannot be executed

原因：n8n の必須入力不足（例：Message Text が空）。

対応：【⑥】Message Text を入れて再実行。

想定外の時間に動く

原因：Schedule Trigger に複数ルール（OR）を入れている。

対応：【⑦】不要ルールを削除し、テスト／本番を分ける。

用語メモ（読み返し用） #

OAuth Scope

API を呼ぶ権利（入口の鍵）。実際の可否はチャンネル参加状態などにも依存。

Bot User OAuth Token（xoxb-）

n8n が Slack を操作するための鍵。Scope 変更後は再インストールで再発行が必要。

Install / Reinstall to Workspace

Scope 変更を Workspace に反映し、Token を作り直す工程。

Execute と Activate

Execute＝手動テスト。Activate＝スケジュール実行（本番稼働）。

セキュリティ注意Token の取り扱い
  • Bot Token（xoxb-…）は漏洩すると第三者が投稿できる可能性があります。
  • ナレッジ共有時は Token を貼らない／伏字にする運用が安全です。
