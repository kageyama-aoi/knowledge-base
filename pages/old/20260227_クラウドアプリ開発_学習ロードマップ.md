# クラウド環境でアプリを作る 学習ロードマップ

> 「触れる → 動く → つながる」を実装レベルで体験する

---

## 目的

以下の3つのクラウド環境を**実際に触って、動かせる状態になること**：

| サービス | 役割 | 体験内容 |
|---|---|---|
| Vercel | 画面を公開 | デプロイ体験 |
| Supabase | DB・認証 | データ保存体験 |
| Cloudflare | エッジ実行 | 高速API体験 |

**最終ゴール：**

> ログインできて、DBに保存できて、公開URLがあるアプリを作る

---

## 全体の進め方

```
① 公開 → ② 保存 → ③ 高速化
```

**この順番の理由：** 成功体験が早い順に進めることで、挫折しにくくなる。

---

## Phase 0：開発環境の準備

### 必須ツールのインストール

- Node.js（LTS版）
- Git
- VS Code

インストール確認：

```bash
node -v
git --version
```

### GitHubアカウントの準備

- アカウントを作成する
- リポジトリを1つ作る（Public で OK）

---

## Phase 1：Vercel で「公開」を体験する

### 1-1. アカウント作成

<https://vercel.com>

→ GitHub連携でログイン

### 1-2. 最速でデプロイする

GitHubに `index.html` を追加：

```html
<h1>Hello Cloud!</h1>
```

Pushした後、Vercelで：

1. Import Project
2. Deploy

→ 数十秒で公開URLが発行される

**チェックポイント：** 「世界に公開」を体験できた

---

## Phase 2：Supabase で「DB・認証」を体験する

### 2-1. アカウント作成

<https://supabase.com>

→ GitHubでログイン

### 2-2. プロジェクトを作成する

- 名前：`demo-app`
- リージョン：Tokyo

### 2-3. テーブルを作成する

Table Editor → New table で `messages` テーブルを作成：

| カラム名 | 型 | 設定 |
|---|---|---|
| id | uuid | デフォルト |
| text | text | 必須 |
| created_at | timestamp | `now()` |

### 2-4. APIキーを取得する

Project Settings → API から以下を控える：

- Project URL
- anon public key

### 2-5. フロントエンドから接続する

React（Vite）プロジェクトを作成：

```bash
npm create vite@latest
npm install
npm install @supabase/supabase-js
```

Supabase接続コード：

```js
import { createClient } from '@supabase/supabase-js'

const supabase = createClient(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY
)
```

`.env` ファイルに記載：

```
VITE_SUPABASE_URL=xxxx
VITE_SUPABASE_ANON_KEY=xxxx
```

Vercel側の環境変数にも同じ値を登録する。

**チェックポイント：**

- テキストを入力する
- DBに保存される
- 再表示できる

→ 「BaaSを使ったWebアプリ開発」の体験完了

---

## Phase 3：Cloudflare で「エッジ実行」を体験する

### 3-1. アカウント作成

<https://dash.cloudflare.com>

### 3-2. Workerを作成してデプロイする

Workers → Create から以下のコードをデプロイ：

```js
export default {
  async fetch(request) {
    return new Response("Hello from Edge!")
  }
}
```

→ URLが発行される

**体験ポイント：**

- サーバー契約不要
- ほぼ即時デプロイ
- 世界中で実行される

これが「Edge実行」の感覚。

---

## Phase 4：3つを組み合わせる

### 最小構成のアーキテクチャ

```
Vercel      → 画面（フロントエンド）
Supabase    → DB（データ保存）
Cloudflare  → Webhook処理（バックグラウンド処理）
```

### 組み合わせの例：フォーム → Slack通知

1. フォームに入力する
2. Supabaseにデータを保存する
3. Cloudflare Workerに通知する
4. Slackにメッセージが届く

## 概念メモ

| 用語 | 意味 |
|---|---|
| Serverless | サーバー管理が不要な実行環境 |
| Edge | ユーザーに近い場所で処理を実行すること |
| BaaS | バックエンド機能を外部サービスとして利用すること |
| CI/CD | コードのPushから自動テスト・自動デプロイまでの仕組み |

## なぜこの順番なのか

1. **まず公開**（達成感を得やすい）
2. **次にデータ**（アプリの本質）
3. **最後に高速化**（応用・最適化）

逆の順番にすると難易度が急上昇する。
