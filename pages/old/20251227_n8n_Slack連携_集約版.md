
## 基礎知識：Slack認証の仕組み

### Token と Scope

- **Token** = Slackが「このリクエストは誰が何の権限でやってるか」を判断する鍵
- **Scope** = Tokenに紐づく「できることの範囲」（メッセージ投稿、チャンネル閲覧 等）
- **Bot Token** = BotとしてSlack上で動作するための鍵

Token と Scope を理解すると、Slack連携のトラブル時に「どこで詰まっているか」を切り分けやすくなる。

### アプリの使い分け

| パターン | メリット | デメリット |
|---|---|---|
| 既存アプリを使う | 管理が一元化できる | 権限が肥大化しがち |
| 新規アプリを作る | 用途別に最小権限にできる | 管理対象が増える |

## パターン1：Slack → n8n Webhook（基礎・指示トリガー型）

### 概要

Slackに特定のメッセージやボタン操作 → n8n Webhookが受信 → Python/GASを実行

### Slack App作成手順

1. [api.slack.com/apps](https://api.slack.com/apps) でアプリ作成
2. **Incoming Webhooks** または **Slash Commands** を有効化
3. Bot Token Scopesに必要なスコープを追加（最低: `chat:write`）
4. ワークスペースにインストール
5. Bot User OAuth Tokenを取得して n8n の Credential に登録

### n8n Webhookノード設定

| 項目 | 値 |
|---|---|
| HTTP Method | POST |
| Path | `slack-trigger`（任意） |
| Authentication | None（まずは最小構成） |
| Respond | Immediately |

### Slack → Python実行（レベル1〜5の考え方）

| レベル | 内容 |
|---|---|
| 1 | Slackメッセージ受信→n8n Webhook疎通確認 |
| 2 | メッセージ内容でPython実行を分岐 |
| 3 | 実行結果をSlackに返す |
| 4 | エラーハンドリング・タイムアウト対応 |
| 5 | 本番環境化・認証強化 |

## パターン2：Slack Bot / Event Subscriptions → n8n

### 概要

Slackのイベント（メンション・リアクション等）を n8n で受信して処理する。

### Slack App設定（Event Subscriptions）

1. **Event Subscriptions** を有効化
2. Request URLに n8n の Webhook URL を貼る
3. **challenge応答が必要**: n8n の Webhook は `challenge` に対して即座にレスポンスを返すこと
4. Subscribe to bot eventsに必要なイベントを追加
   - `message.channels`（チャンネルのメッセージ）
   - `app_mention`（Botへのメンション）
   - `reaction_added`（リアクション）など

### challenge応答（n8n側の設定）

Slack は Event Subscriptions 設定時にチャレンジリクエストを送ってくる。
n8n は `Respond with JSON` で `{ "challenge": "{{$json.body.challenge}}" }` を返す必要がある。

```json
// Slack が送ってくるリクエスト
{
  "token": "...",
  "challenge": "3eZbrw1aBm2rZgRNFdxV2595E9zY3udr",
  "type": "url_verification"
}

// n8n が返すべきレスポンス
{
  "challenge": "3eZbrw1aBm2rZgRNFdxV2595E9zY3udr"
}
```

### イベント受信後の処理例

```
Webhook（イベント受信）
  ↓
IF（type === "app_mention"かチェック）
  ↓ True
Set（必要なフィールドを抽出）
  ↓
Slack（返信投稿）
```

## パターン3：n8n × Slack × GAS（実装版）

### 設計思想

- Slack側からボタンを押すだけで GAS を実行する仕組み
- 「設定値を固定し、壊れにくく、後から再開できる形」を目指す
- Slack payload をそのまま信じない → n8n側で一度JSONとして正規化する

### Interactive Components（ボタン）の利用

1. Slack App で **Interactivity & Shortcuts** を有効化
2. Request URL に n8n の Webhook URL を設定
3. Slack に Block UI（ボタン）を投稿

**Block UI（ボタン投稿）例:**
```json
{
  "blocks": [
    {
      "type": "actions",
      "elements": [
        {
          "type": "button",
          "text": { "type": "plain_text", "text": "実行する" },
          "action_id": "run_gas",
          "value": "run_gas"
        }
      ]
    }
  ]
}
```

### n8n フロー

```
Webhook（Slack Interactive payload受信）
  ↓
Code Node（payload を JSON.parse → action を抽出）
  ↓
IF（action値で分岐）
  ↓ True
HTTP Request（GAS Web App URLへPOST）
  ↓
Slack（結果を返信）
```

**Code Nodeでのpayload正規化例:**

```javascript
const payloadStr = $json.body.payload;
const payload = JSON.parse(payloadStr);
const action = payload.actions?.[0]?.action_id ?? '';

return [{ json: { action, user: payload.user.name, channel: payload.channel.id } }];
```

### GAS側（Web App）

```javascript
// doPost で n8n からのリクエストを受け取る
function doPost(e) {
  const data = JSON.parse(e.postData.contents);
  // 処理...
  return ContentService
    .createTextOutput(JSON.stringify({ status: 'ok' }))
    .setMimeType(ContentService.MimeType.JSON);
}
```

## 詰まりポイントまとめ

| 問題 | 原因 | 解決 |
|---|---|---|
| Event Subscriptionsの設定が保存できない | challenge応答ができていない | n8n側で challenge をそのまま返す |
| ボタンを押してもn8nに届かない | Interactive ComponentsのRequest URLが未設定 | Slack App設定でURL登録 |
| Slack APIエラー `not_in_channel` | BotがチャンネルにJoinしていない | Botをチャンネルに招待 |
| payload の中身が取れない | Interactive payloadはURLエンコードされた文字列 | `JSON.parse($json.body.payload)` で変換 |
| GAS Web Appが応答しない | スクリプトが「全員」で公開されていない | デプロイ設定を確認 |
| n8nのWebhookが届かない（ローカル） | localhostは外部から届かない | ngrokで公開URLを作る |

## セキュリティ（後から強化する順番）

1. **Signing Secret の検証**: Slack からのリクエストが本物か確認する
2. **Basic認証 / Token認証**: n8n Webhookへの不正アクセスを防ぐ
3. **最小Scope**: 必要なスコープだけに絞る
4. **本番URLへの移行**: ngrok → 固定のサーバー/Cloudflare Tunnel等
