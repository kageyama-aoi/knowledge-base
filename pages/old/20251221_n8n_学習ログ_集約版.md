## n8n って何？

**ノーコード/ローコードの自動化ツール。**

「ある出来事をきっかけに（トリガー）」→「処理をつないで（ワークフロー）」→「別の場所へ結果を渡す」ための実行基盤。

**強いポイント:**
- UIで組める（ノードをつなぐ）＋必要なら Code Node（JS）でロジックを書ける
- Webhook、API、Slack、GitHub、GAS、Notionなど外部サービス連携が中心

## Day 1：環境構築（Docker + Gmail OAuth + OpenAI）

### Docker Compose で n8n を立ち上げる

**構成:** n8n本体 ＋ PostgreSQL（永続化DB）

```
① Docker Desktop を起動
② docker compose up -d
③ http://localhost:5678 にアクセス
④ n8n の初期画面が表示されれば成功
```

- n8n と PostgreSQL を一発でまとめて起動できる
- 本番（VPS）移行時も同じ構成を流用できる
- PostgreSQL はホストにポート公開せず、Docker内ネットワークで接続

**重要な環境変数:**

| 変数 | 用途 |
|---|---|
| `GENERIC_TIMEZONE=Asia/Tokyo` | 実行時刻・ログ・スケジュールのズレ防止 |
| `DB_TYPE=postgresdb` | ワークフロー情報の保存先 |
| `N8N_ENFORCE_SETTINGS_FILE_PERMISSIONS=true` | 設定ファイル権限チェック |

### Gmail アカウント連携（OAuth）

n8nは「自前アプリ」として Google に身元登録が必要。
Gmail のID/パスワードを直接渡さないための仕組みが OAuth。

**手順:**
1. Google Cloud プロジェクト作成
2. Gmail API を有効化
3. OAuth クライアントID を作成
4. 承認済みリダイレクト URI を登録: `http://localhost:5678/rest/oauth2-credential/callback`
5. OAuth 同意画面を設定
6. テストユーザーに自分の Gmail を追加

**よくあるエラー:**
- `redirect_uri_mismatch` → URL の完全一致が必要
- `access_denied` → テストユーザー未登録が原因

### OpenAI API キー取得

1. OpenAI のアカウント作成/ログイン
2. API Keys 画面から新規キー発行
3. キーを n8n の Credential に登録
- ※ APIキーは一度しか表示されないため、安全な場所に保管
- ※ Billing を設定しない限り無料枠内で使える

**Day 1 の学びポイント:**
- Dockerのエラーはホスト環境が原因なことが多い（5432競合はWindows側PostgreSQLが原因）
- n8n は SaaS ではなく Web サービス → OAuth 設定が必要になる理由が理解できた
- OAuthのエラーは段階的に前進している証拠（失敗ではなくチェックポイント）

## Week 1 総まとめ：学びの全体像

学びの流れ：**環境を作る → 概念が分かる → 連携したくなる → 実装が必要になる → 運用が重要になる**

### 基本概念（最重要）

データが流れている理解がないと、WebhookもSlackもGitHubも「なぜ動かないか分からない」状態になる。

| 要素 | 役割 |
|---|---|
| Trigger | 開始点（Webhook / Cron / アプリイベント / 手動実行） |
| Node | 処理（取得・変換・分岐・送信など） |
| Connection | 前ノードの出力を次ノードへ渡す線 |

- 各ノードは基本的に **items（配列）** を扱う
- Expression: `{{$json.xxx}}` で値を埋め込む

### Webhook（発行側 vs 受け側）

| 視点 | 特徴 |
|---|---|
| 発行する側 | 設定はラク（URL貼る＋イベント選ぶ程度） |
| 受け取る側 | URLを用意＋データ形式に合わせた処理＋セキュリティ考慮が必要 |

n8n の Webhook Trigger = 「外から叩ける入口」
入口ができると、Slack通知、GitHub操作、DB書き込み等へ繋げられる。

### Slack連携（Token / Scope / Bot）

- **Token** = 「このリクエストは誰が何の権限でやってるか」を判断する鍵
- **Scope** = Token に紐づく「できることの範囲」
- **Bot** = Slack上で動作する "アプリの人格"

アプリの使い分け判断軸:
- **既存アプリ活用**: 管理が一元化できるが、権限が肥大化しがち
- **新規アプリ作成**: 用途別に最小権限にできるが、管理対象が増える

### GitHub連携（Issue / コメント / ラベル）

- Issue: 「やること」「バグ」「改善案」をチケット化したメモ兼タスク
- n8nと繋ぐと: Slack投稿→Issue作成、特定キーワード→ラベル付け、定型返信→コメント追加

### Code Node（JavaScript）

UIだけでは面倒な「整形」「抽出」「条件分岐」「集計」をやる場所。

実例: 天気JSONから最高/最低を取り出してSlackに投げる
- どの階層に値があるか
- 配列/オブジェクトの扱い
- n8nの「items構造」を崩さず返す

### n8n × GAS 連携の設計思想

得意領域を分担させる:
- **GAS**: Google側（スプレッドシート/Drive/カレンダー等）の操作が得意
- **n8n**: 外部連携のハブ＋ワークフロー可視化が得意

例:
- `Slack → n8n → GAS → スプレッドシート更新`
- `スプレッドシート更新 → n8n → Slack通知`
